# Rust → Wasm 垂直基建编译与边界规范 (Vertical Base Spec) V2.0

## 0. 核心哲学与不可动摇的铁律

本规范是"唯物辩证主义与截拳道"架构哲学在编译与边界层的工程化落地。它基于以下物理事实：

- **Wasm 线性内存只增不减**，必须内部回收。
- **JS ↔ Wasm 边界是昂贵的 RPC**，任何隐式转换都是性能负债。
- **浏览器主线程不容阻塞**，必须顺应事件循环。
- **零拷贝视图是 unsafe 操作**，生命周期必须受规约约束。

由此衍生四条铁律：

1. **[IRON-01] 体积即王道**：编译产物必须被极致压缩，所有调试信息、展开表、冗余段全部剥离。
2. **[IRON-02] 边界零拷贝**：高频数据路径上，永远传递指针与长度，绝不进行序列化；零拷贝视图的生命周期必须在文档中标注。
3. **[IRON-03] 内存分治**：全局状态静态驻留，帧级对象随帧湮灭，杜绝动态分配碎片。
4. **[IRON-04] 跨源隔离可见即所得**：凡启用 `SharedArrayBuffer` 必须文档化 COOP/COEP 头配置要求。

后续所有规则均是对这四条铁律的展开。

## 1. Stage: 编译与产物控制 (Artifact & Build Control)

### 1.1 发布配置 (MUST)

在 `Cargo.toml` 中，必须为 Wasm 构建使用如下发布配置，且不得随意覆盖：

```toml
[profile.release]
opt-level = "z"       # 体积优先（或 "s"，若性能实测差异显著）
lto = true            # 强制 fat LTO，最大化死代码消除
codegen-units = 1     # 单一编译单元，使优化器获得全局视野
panic = "abort"       # 禁用栈展开，减少体积并避免浏览器内展开的未定义行为
strip = true          # 剥离符号，进一步压缩
````

**依据**：Leptos、NEAR、SWC 等主流项目均以此为基础配置，经验证可有效缩小二进制。

### **1.2 后处理优化 (MUST)**

构建流水线的最后一步，必须使用 `wasm-opt`（Binaryen 工具集 v110+）执行二进制级别优化：

bash

```
wasm-opt -Oz --enable-simd -o output.wasm input.wasm
```

- `-Oz` 为终极体积优化，通常可额外缩减 15-20%。
- `--enable-simd` 可选，需根据目标浏览器兼容性决策。
- 必须在 CI 中验证 `wasm-opt` 后文件大小回归不超标。

### **1.3 分配器替换 (SHOULD)**

全局分配器的选择直接影响 .wasm 基线体积：

- **弃用警告**：`wee_alloc` 已长期不维护，**新项目禁止选用**。
- **推荐**：`talc`（现代、活跃维护）或 `MiniAlloc`（专为 Wasm 优化，利用零填充页减少内存占用）。
- **收益**：默认分配器约 10KB，替换后可降至 \~1KB，是缩小基线体积最直接的手段。

决策记录需注明所选分配器的体积基准测试结果。

## **2. Stage: FFI 与跨语言边界 (JS-Wasm Boundary)**

### **2.1 零拷贝内存视图 (MUST)**

处理音频、视频、图像像素或大型矩阵时，必须使用零拷贝内存视图，而非任何序列化方案。

**安全封装模板**（必须遵循）：

rust

```
use wasm_bindgen::prelude::*;

/// 零拷贝视图的安全封装。
/// # Safety
/// - 调用者必须确保 `ptr` 和 `len` 来自有效的 Wasm 线性内存。
/// - JS 侧持有的 TypedArray 视图必须在本 tick 内消费，不得跨帧保存。
#[wasm_bindgen]
pub struct WasmSlice {
    ptr: *const u8,
    len: usize,
}

#[wasm_bindgen]
impl WasmSlice {
    pub fn ptr(&self) -> *const u8 { self.ptr }
    pub fn len(&self) -> usize { self.len }

    /// 从 Rust 切片创建零拷贝视图。
    /// 调用者负责确保切片在原内存中存活足够长。
    pub fn from_slice(data: &[u8]) -> WasmSlice {
        WasmSlice {
            ptr: data.as_ptr(),
            len: data.len(),
        }
    }
}
```

**JS 侧强制消费协议**：

javascript

```
const view = wasm.get_frame_data();
const buffer = new Float64Array(wasm.memory.buffer, view.ptr(), view.len() / 8);
// 在本 tick 内立即消费，不存储到闭包、异步回调或全局变量
gl.bufferData(gl.ARRAY_BUFFER, buffer, gl.DYNAMIC_DRAW);
```

### **2.2 边界安全风险**

- **悬垂指针风险**：若 Rust 侧释放了堆数据，而 JS 侧仍持有 TypedArray 视图，将读取无效内存或导致崩溃。Rust 所有权模型无法跨越 FFI 边界自动保护，必须通过规范约束生命周期。
- **生命周期约定**：所有零拷贝视图的文档注释中必须声明 `# Safety` 一节，明确视图的有效期（通常"本帧内有效"或"至下次 reset 前有效"）。

### **2.3 边界类型的显式契约 (MUST)**

所有暴露给 JS 的结构体必须使用 `#[wasm_bindgen]` 宏，参数必须是明确的标量或 `WasmSlice`。

rust

```
// ❌ 禁止：依赖隐式序列化跨界
#[wasm_bindgen]
pub fn process_data(data: &JsValue) { ... }

// ✅ 正确：显式标量传递
#[wasm_bindgen]
pub fn process_config(flag: u32, threshold: f64) { ... }

// ✅ 正确：零拷贝大数据路径
#[wasm_bindgen]
pub fn get_large_buffer() -> WasmSlice { ... }
```

## **3. Stage: 内存与生命周期 (Linear Memory Lifecycle)**

### **3.1 生命周期分离 (MUST)**

应用状态必须严格区分为两类生命周期：

**全局驻留 (Static/Long-lived)**：

- 核心业务状态机（如 `AppState`）使用预分配 `Vec`、`arrayvec::ArrayVec` 或 `SlotMap`。
- 驻留在 `static` 或 `wasm-bindgen` 导出的根对象中，避免运行时扩容。

**帧级生灭 (Per-frame/Transient)**：

- UI 渲染中间变量、虚拟 Diff 树等必须使用 Arena 分配器（如 `bumpalo`）。
- 每帧入口第一行代码：`FRAME_ARENA.with(|a| a.borrow_mut().reset());`
- **安全警告**：`bumpalo` 的 bump 分配默认不守护内存安全，不得存储带有 `Drop` 复杂实现的对象到 Arena 中，除非确认其行为。
- **性能注意**：Arena 在特定场景下可能不总是比堆分配快，建议以真实帧负载做基准测试。

### **3.2 内存泄漏的物理防御 (MUST)**

- JS 组件销毁时，必须显式调用 Wasm 对象的 `.free()` 方法。
- 探索启用 `WeakRef`（ES2021）作为自动化清理的辅助手段，但不作为唯一依赖。
- 在页面卸载事件 (`beforeunload`) 中执行全局清理序列。

## **4. Stage: 并发与事件驱动 (Concurrency & Event Loop)**

### **4.1 异步模型的无缝映射 (MUST)**

所有 I/O 操作必须使用 `wasm-bindgen-futures`，该 crate 提供 `JsFuture`（Promise → Future）和 `future_to_promise`（Future → Promise）双向转换。

rust

```
use wasm_bindgen_futures::spawn_local;

#[wasm_bindgen]
pub fn start_network_task() {
    spawn_local(async {
        let result = fetch_data().await;
        update_state(result);
    });
}
```

### **4.2 严禁阻塞型 API (MUST NOT)**

- 禁止：`std::thread::sleep`、`std::sync::Mutex::lock()`（竞争时冻结标签页）
- 禁止：任何形式的忙等待循环
- 单次 Wasm 函数调用超过约 5ms 应考虑分片或移交 Worker

### **4.3 重度计算的 Worker 隔离 (SHOULD)**

计划使用 Worker + `SharedArrayBuffer` 前，必须评估以下前置条件：

1. **COOP / COEP 服务端头配置**：`SharedArrayBuffer` 要求页面处于"跨源隔离"状态，服务端必须设置：
   - `Cross-Origin-Opener-Policy: same-origin`
   - `Cross-Origin-Embedder-Policy: require-corp`
   - **若部署在 GitHub Pages 等不支持自定义 HTTP 头的平台，需额外使用 Service Worker 注入**，但此方案兼容性受限，必须在架构决策记录中注明。
2. 共享内存布局必须使用 `#[repr(C)]` 结构体定义，字段偏移明确可计算。
3. 多 Worker 间同步必须使用 `std::sync::atomic`（`AtomicU32`、`AtomicU64`），结合 `Atomics.wait` / `Atomics.notify`。
4. **备选方案**：若 COOP/COEP 不可行，使用 `postMessage` 传递所有权（Transferables）是降级方案，而非零拷贝。

## **5. Stage: 通用代码规范的 Wasm 适配**

### **5.1 错误处理 (Error Handling) - MUST**

跨 FFI 边界的 `Result<T, E>` 必须转换为 `Result<T, JsValue>`，建议使用 `thiserror` 派生 `Error` trait：

rust

```
use thiserror::Error;

#[derive(Error, Debug)]
pub enum AppError {
    #[error("invalid command: {0}")]
    InvalidCommand(u32),
    #[error("state conflict")]
    StateConflict,
}

#[wasm_bindgen]
pub fn execute_command(cmd: u32) -> Result<(), JsValue> {
    core_logic(cmd).map_err(|e| JsValue::from_str(&e.to_string()))
}
```

**关于 **`panic = "abort"`** 下的恢复**：Cloudflare Workers 的经验表明，`wasm-bindgen` 将 panic 视为不可恢复，panic 后整个 Wasm 实例被视为无效状态。因此，业务逻辑必须用 `Result` 而非 `panic!` 处理可预见的失败路径。

### **5.2 日志与可观测性 (Logging) - MUST**

rust

```
// 在 Wasm 模块初始化时，必须调用以下双重配置：
use console_error_panic_hook;
use tracing_wasm;

#[wasm_bindgen(start)]
pub fn init() {
    // 步骤1：将 Rust panic 转换为格式化的控制台消息，避免收到无法阅读的
    // "RuntimeError: unreachable executed" 错误
    console_error_panic_hook::set_once();

    // 步骤2：配置 tracing，将结构化日志输出到浏览器控制台
    // 注意：需验证所使用的 tracing-wasm 版本与 tracing 生态的兼容性
    tracing_wasm::set_as_global_default();
}
```

**依据**：`console_error_panic_hook` 将 panic 消息转发到 `console.error`；`tracing-wasm` 曾与旧版 `tracing` API 绑定，选型时需确认版本兼容性。

## **6. 禁止清单 (Hard Forbidden)**

- **\[F-01]** 不得在 `wasm32-unknown-unknown` 目标下使用 `std::thread::sleep` 或阻塞型 `Mutex::lock()`。
- **\[F-02]** 不得在高频 FFI 路径上调用 `serde_json` 或任何序列化库。
- **\[F-03]** 不得在帧循环内执行 `Box::new`、`Vec::push`（触发堆分配）除非在全局状态预分配容器中。
- **\[F-04]** 不得将 `println!` 用于生产日志，必须使用 `tracing` + `console_error_panic_hook`。
- **\[F-05]** 不得依赖 Wasm GC 提案（参考类型等），至 Wasm 2.0 规范落地且主流浏览器兼容后再引入。
- **\[F-06]** 不得选用 `wee_alloc` 作为新项目的分配器（已不再维护）。
- **\[F-07]** 不得在未文档化 COOP/COEP 配置要求的情况下声称支持 `SharedArrayBuffer`。

***

- `Cargo.toml` 中 `[profile.release]` 包含 `opt-level="z"`, `lto=true`, `codegen-units=1`, `panic="abort"`, `strip=true`。
- CI 流水线集成 `wasm-opt -Oz` 步骤，且有文件大小回归检查。
- 分配器已从默认替换为 `talc` 或 `MiniAlloc`（禁用 `wee_alloc`），体积基准测试完成。
- 所有跨边界大数据路径使用零拷贝视图封装，且生命周期在文档中明确标注。
- FFI 接口全部使用标量参数或 `WasmSlice`，未出现 `JsValue` 透传。
- 帧循环入口第一行代码为 Arena `reset()`。
- 异步 I/O 仅使用 `wasm-bindgen-futures`，无阻塞调用。
- 若启用 `SharedArrayBuffer`，COOP/COEP 头配置方案已写入架构决策记录。
- 初始化代码中调用了 `console_error_panic_hook::set_once()` 和 `tracing_wasm::set_as_global_default()`。
- 错误处理使用 `Result<T, JsValue>` 模式，无误用 `panic!` 处理业务逻辑。
